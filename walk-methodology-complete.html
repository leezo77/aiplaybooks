The "Walk Before You Run" Philosophy

The goal of this methodology is to get a valuable, safe, and measurable agent into a controlled production environment. We focus on four core phases: Discover, Decide, Make, and Validate. Perfect for first-time builders, 2-4 week delivery cycles, and immediate ROI.

PART 1: WALK PLAYBOOK

1. Discover

Goal: Start here to clarify goals, identify risks, and ensure your agent is solving a real, measurable problem. A strong foundation prevents wasted effort later.

Problem Framing & Success Criteria

A structured canvas to ensure your agent solves a concrete, high-value problem with measurable success.

What it is

A simple, one-page canvas to define a concrete problem and how to measure success. This is the most critical step.

When to use

At the very beginning of the project, before any technical decisions are made.

MVP Steps

  1. Start Small, Scale Later: Scope creep is the enemy of effective AI. Define the smallest possible task the agent can perform to demonstrate value.
  2. Define the User and Outcome: State the primary user and the desired outcome in a simple format: verb + object + constraint (e.g., "Categorize product SKUs from an email order").
  3. Define Inputs and Allowed Actions: List what the agent receives (e.g., email text) and what it is allowed to do (e.g., call the product database API).
  4. Draft Success Metrics: Define clear, measurable goals. Focus on Action Completion (AC) and Tool Selection Quality (TSQ). Set targets for latency and cost per action.

Outputs

A simple 1-page document with a clear goal statement, defined inputs/actions, and success metrics.

Pitfalls to Avoid

  • Starting by picking a cool AI model instead of defining the problem
  • Having vague success criteria like "improve efficiency"
  • Giving the agent unlimited powers or tool access

Risk & Harm Mapping

A quick exercise to identify the most obvious potential harms and plan basic mitigations.

What it is

A quick exercise to identify the most obvious potential harms (e.g., privacy leaks, incorrect actions) and plan basic mitigations.

When to use

Immediately after problem framing.

MVP Steps

  1. Map Data Sensitivity: Identify if the agent will handle sensitive data (PII, PHI).
  2. List Attack Surfaces: Note the primary ways the agent could be misused (malicious user inputs, insecure tool endpoints).
  3. Define Basic Mitigations: For each major risk, define a simple mitigation strategy (e.g., "add an input validation step," "require human approval for actions that delete data").

Outputs

A simple risk register listing the top 3-5 risks and their planned mitigations.

Pitfalls to Avoid

  • Treating safety as something to add later
  • Not considering the security of the tools the agent uses

2. Decide

Goal: Choose the right technical path. This phase is about selecting the simplest architecture and data strategy that meets the goals you defined in the Discover phase.

Pattern Selection (Simplified)

A decision tree to choose the simplest agent architecture that will solve your defined problem.

When to use

After problem framing and risk mapping.

MVP Steps

Workflow-Embedded Agent

Use this if the task is one step in an existing process, needs a fast response, and uses 1-2 tools. This is the simplest pattern, ideal for low-latency tasks.

Independent Agent Service

Choose this if multiple applications need to call the agent. This provides a reusable API for your agent but is more complex.

OutSystems Implementation

AI Agent Builder vs. Agent Workbench

Use AI Agent Builder if: The agent's primary function is information retrieval (RAG), document classification, data extraction, or simple Q&A. This is a no-code solution ideal for low-to-medium complexity use cases.

Use Agent Workbench if: The agent's core function involves more complex reasoning, multi-step processes, or performing actions via tool integration. This is a low-code environment for building sophisticated, enterprise-grade agents.

Outputs

A clear decision on the architectural pattern with a brief rationale.

Pitfalls to Avoid

Over-engineering with a multi-agent system when a single agent will do; tightly coupling the agent to a single application if it needs to be reused.

Grounding Data Strategy (RAG Basics)

A basic plan for how your agent will use internal knowledge reliably using Retrieval-Augmented Generation.

When to use

If your agent's answers or actions must be based on your company's internal or up-to-date data.

MVP Steps

  1. Inventory Data Sources: List the 1-3 most critical data sources the agent needs.
  2. Choose a Chunking Strategy: Start with a simple fixed-size chunking method. Semantic chunking is better but adds complexity for an MVP.
  3. Select a Vector Store: For an MVP, a simple in-memory store like Chroma or a managed service like Pinecone can get you started quickly with minimal operational overhead.
  4. Plan a Basic Data Pipeline: Outline the simple flow: clean source data → embed → index in vector store.

OutSystems Implementation

Data Fabric

The OutSystems Developer Cloud (ODC) provides Data Fabric, a unified, low-code layer for accessing different enterprise data sources like databases and APIs. For RAG agents, Data Fabric can provide a secure and centralized connection point for the agent's data retrieval step, simplifying integration and enforcing data governance.

Outputs

A basic RAG architecture diagram and choice of vector store.

Pitfalls to Avoid

  • Trying to connect all company data at once
  • Using fixed-size chunks for documents where context is easily lost
  • Not having a plan to keep the indexed data fresh

3. Make

Goal: Implement the agent using production-ready practices. This is where you leverage OutSystems' capabilities to build a robust and governable agent, not just a demo.

Agent Skeleton (MVP Checklist)

A production-ready starting point that includes essential guardrails and observability from day one to avoid building fragile "demo-ware".

When to use

At the start of implementation.

The MVP Checklist

  • Typed Schemas: Use a tool like JSON Schema to define structured inputs and outputs. This is the foundation of a reliable agent.
  • Tool Adapters with Error Handling: Create clear interfaces for any tools the agent uses. Ensure they handle errors gracefully and can retry if necessary.
  • Basic Guardrails: Implement input sanitation to block malicious prompts and output validators to ensure the agent's response is in the correct format.
  • Simple Memory: For conversational agents, implement a basic short-term memory that stores the last few turns of the conversation. Long-term memory can be added later.
  • Observability Hooks: Integrate basic logging and tracing from day one. Use a standard like OpenTelemetry to log key events: agent start/end, tool calls, and final output.
  • Human-in-the-Loop (for risky actions): If the agent can perform irreversible actions, build a mandatory "escape hatch" where it must pause and get human approval before proceeding.
  • A Kill Switch: Implement a simple mechanism, like a feature flag, to quickly and completely disable the agent's automated actions in production if something goes wrong.

OutSystems Implementation

Agent Workbench AgentTask

An agent built with the OutSystems Agent Workbench is implemented as a visual Server Action called an AgentTask. This action orchestrates the agent's reasoning loop through a sequence of granular steps, making the logic transparent and maintainable. A typical MVP sequence includes:

  • GetGroundingData: Fetches contextual information from data sources, often via Data Fabric.
  • BuildMessages: Constructs the final prompt for the LLM by combining the user query, retrieved data, and conversation history.
  • CallLLM: Sends the prompt to the selected AI model (e.g., Azure OpenAI).
  • StoreMemory: Persists the conversation turn to the OutSystems database to enable memory.

Outputs

A scaffolded agent in your OutSystems app, ready for logic implementation that includes all the checklist items.

Pitfalls to Avoid

  • Hard-coding prompts in the logic; no error handling; missing a kill switch
  • Mixing business logic into prompts
  • Not having typed inputs/outputs
  • Having no way to debug what the agent did

Tools

OutSystems Agent Builder, Agent Workbench for testing and iteration, OutSystems Service Studio, and the Feature Flags Forge component for the kill switch.

4. Validate

Goal: Evaluate the agent's reliability and safety. Before going live, you must have a repeatable way to prove the agent does what it's supposed to do—and doesn't do what it shouldn't.

Agent Evaluation (Core Metrics)

A repeatable process to measure if your agent is successfully achieving its goal.

When to use

Before launch and every time you change the prompt or model.

MVP Steps

  1. Author 10-20 Key Scenarios: Write a set of test cases that represent the agent's most common and important tasks.
  2. Measure Action Completion (AC): This is your primary success metric. It's a simple question: did the agent successfully achieve the user's end-to-end goal in the scenario?
  3. Measure Tool Selection Quality (TSQ): This is your primary efficiency metric. Did the agent use the correct tools with the correct parameters, and did it avoid unnecessary calls?
  4. Track Latency and Cost: For each scenario, record how long the agent took and how much it cost (e.g., token usage).
  5. Gate your deployment pipeline: Don't deploy if success metrics drop.
Acceptance Criteria Example
  • AC ≥ 85% on key scenarios
  • TSQ ≥ 90%
  • p95 latency ≤ 5 seconds

OutSystems Implementation

ODC Monitoring

You can use the built-in observability tools in the OutSystems Developer Cloud (ODC) to track your MVP's performance against your defined metrics.

  • App Analytics Dashboard: The native dashboard in the ODC Portal monitors key metrics for AI agents, including total requests, response time (latency), and errors.
  • Traces and Logs: For failed scenarios, you can use the ODC Portal to examine distributed traces and logs. Traces capture the sequence of events in an agent's execution, helping you pinpoint the exact step where an issue occurred.

Outputs

A simple test results dashboard; pass/fail criteria for deployments.

Pitfalls to Avoid

  • Only checking if the text output "looks good"; not measuring end-to-end success
  • Not measuring end-to-end success

Tools

OutSystems AI Agent Inspector for debugging; a simple OutSystems app to run scenarios and log results.

Basic Safety Testing

A focused set of tests for the most likely security and safety risks identified during the Discover phase.

When to use

Before the first production deployment.

MVP Steps

  1. Prompt Injection Tests: Use a library of known prompt injection attacks to test if the agent can be forced to ignore its instructions.
  2. Data Leakage Tests: If the agent handles sensitive data, create tests to see if you can trick it into revealing that data.
  3. Validate Guardrails: Test the input and output validators you built in the "Make" phase to ensure they work as expected.

Outputs

A safety scorecard showing pass/fail for the essential safety checks.

Pitfalls to Avoid

  • Treating safety as an advisory check
  • Not having automated safety tests that can be run regularly

Getting Started with Walk Use Cases

Choose Your First Agent:

  • Start with a high-volume, repetitive task
  • Pick something with clear success criteria
  • Focus on single-domain problems

Implementation Timeline:

  • Week 1-2: Problem framing and data preparation
  • Week 2-3: Agent development and testing
  • Week 3-4: Integration and deployment
  • Week 4+: Monitor, measure, and iterate

Quick Reference: The MVP Mindset

  • Start with the smallest, most specific problem
  • Choose the simplest architecture that works
  • If the agent needs knowledge, use basic RAG
  • Build a stable skeleton with logging and kill switch
  • Prove its value with Action Completion (AC) and Tool Selection Quality (TSQ)
