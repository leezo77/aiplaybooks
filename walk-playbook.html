Walk Before You Run

This playbook provides the essential, minimum viable steps to build your first reliable AI Agent using OutSystems. Focus on these core methods to deliver value quickly and safely, establishing a solid foundation before scaling to more complex "Run" scenarios.

The "Walk Before You Run" Philosophy

The goal of this playbook is to get a valuable, safe, and measurable agent into a controlled production environment. We focus on four core phases: Discover, Decide, Make, and Validate.

Discover

Start here to clarify goals, identify risks, and ensure your agent is solving a real, measurable problem. A strong foundation prevents wasted effort later.

Problem Framing & Success Criteria

A structured canvas to ensure your agent solves a concrete, high-value problem with measurable success.

What it is

A simple, one-page canvas to define a concrete problem and how to measure success. This is the most critical step.

When to use

At the very beginning of the project, before any technical decisions are made.

MVP Steps
  1. Start Small, Scale Later: Scope creep is the enemy of effective AI. Define the smallest possible task the agent can perform to demonstrate value.
  2. Define the User and Outcome: State the primary user and the desired outcome in a simple format: verb + object + constraint (e.g., "Categorize product SKUs from an email order").
  3. Define Inputs and Allowed Actions: List what the agent receives (e.g., email text) and what it is allowed to do (e.g., call the product database API).
  4. Draft Success Metrics: Define clear, measurable goals. Focus on Action Completion (AC) and Tool Selection Quality (TSQ). Set targets for latency and cost per action.
Outputs

A simple 1-page document with a clear goal statement, defined inputs/actions, and success metrics.

Pitfalls to Avoid
  • Starting by picking a cool AI model instead of defining the problem
  • Having vague success criteria like "improve efficiency"
  • Giving the agent unlimited powers or tool access
Tools

A simple document template; stakeholder interview notes.

Risk & Harm Mapping

A quick exercise to identify the most obvious potential harms and plan basic mitigations.

What it is

A quick exercise to identify the most obvious potential harms (e.g., privacy leaks, incorrect actions) and plan basic mitigations.

When to use

Immediately after problem framing.

MVP Steps
  1. Map Data Sensitivity: Identify if the agent will handle sensitive data (PII, PHI).
  2. List Attack Surfaces: Note the primary ways the agent could be misused (malicious user inputs, insecure tool endpoints).
  3. Define Basic Mitigations: For each major risk, define a simple mitigation strategy (e.g., "add an input validation step," "require human approval for actions that delete data").
Outputs

A simple risk register listing the top 3-5 risks and their planned mitigations.

Pitfalls to Avoid
  • Treating safety as something to add later
  • Not considering the security of the tools the agent uses

Decide

Choose the right technical path. This phase is about selecting the simplest architecture and data strategy that meets the goals you defined in the Discover phase.

Pattern Selection (Simplified)

A decision tree to choose the simplest agent architecture that will solve your defined problem.

When to use

After problem framing and risk mapping.

MVP Steps
Workflow-Embedded Agent

Use this if the task is one step in an existing process, needs a fast response, and uses 1-2 tools. This is the simplest pattern, ideal for low-latency tasks.

Independent Agent Service

Choose this if multiple applications need to call the agent. This provides a reusable API for your agent but is more complex.

OutSystems Implementation
AI Agent Builder vs. Agent Workbench
Use AI Agent Builder if:

The agent's primary function is information retrieval (RAG), document classification, data extraction, or simple Q&A. This is a no-code solution ideal for low-to-medium complexity use cases.

Use Agent Workbench if:

The agent's core function involves more complex reasoning, multi-step processes, or performing actions via tool integration. This is a low-code environment for building sophisticated, enterprise-grade agents.

Outputs

A clear decision on the architectural pattern with a brief rationale.

Pitfalls to Avoid

Over-engineering with a multi-agent system when a single agent will do; tightly coupling the agent to a single application if it needs to be reused.

Grounding Data Strategy (RAG Basics)

A basic plan for how your agent will use internal knowledge reliably using Retrieval-Augmented Generation.

When to use

If your agent's answers or actions must be based on your company's internal or up-to-date data.

MVP Steps
  1. Inventory Data Sources: List the 1-3 most critical data sources the agent needs.
  2. Choose a Chunking Strategy: Start with a simple fixed-size chunking method. Semantic chunking is better but adds complexity for an MVP.
  3. Select a Vector Store: For an MVP, a simple in-memory store like Chroma or a managed service like Pinecone can get you started quickly with minimal operational overhead.
  4. Plan a Basic Data Pipeline: Outline the simple flow: clean source data ‚Üí embed ‚Üí index in vector store.

  • Not having a plan to keep the indexed data fresh

Make

Implement the agent using production-ready practices. This is where you leverage OutSystems' capabilities to build a robust and governable agent, not just a demo.

Agent Skeleton (MVP Checklist)

A production-ready starting point that includes essential guardrails and observability from day one to avoid building fragile "demo-ware".

When to use

At the start of implementation.

The MVP Checklist
  • Typed Schemas:

    Use a tool like JSON Schema to define structured inputs and outputs. This is the foundation of a reliable agent.

  • Tool Adapters with Error Handling:

    Create clear interfaces for any tools the agent uses. Ensure they handle errors gracefully and can retry if necessary.

  • Basic Guardrails:

    Implement input sanitation to block malicious prompts and output validators to ensure the agent's response is in the correct format.

  • Simple Memory:

    For conversational agents, implement a basic short-term memory that stores the last few turns of the conversation. Long-term memory can be added later.

  • Observability Hooks:

    Integrate basic logging and tracing from day one. Use a standard like OpenTelemetry to log key events: agent start/end, tool calls, and final output.

  • Human-in-the-Loop (for risky actions):

    If the agent can perform irreversible actions, build a mandatory "escape hatch" where it must pause and get human approval before proceeding.

  • A Kill Switch:

    Implement a simple mechanism, like a feature flag, to quickly and completely disable the agent's automated actions in production if something goes wrong.

Pitfalls to Avoid
  • Hard-coding prompts in the logic; no error handling; missing a kill switch
  • Mixing business logic into prompts
  • Not having typed inputs/outputs
  • Having no way to debug what the agent did

Validate

Evaluate the agent's reliability and safety. Before going live, you must have a repeatable way to prove the agent does what it's supposed to do‚Äîand doesn't do what it shouldn't.

Agent Evaluation (Core Metrics)

A repeatable process to measure if your agent is successfully achieving its goal.

When to use

Before launch and every time you change the prompt or model.

MVP Steps
  1. Author 10-20 Key Scenarios: Write a set of test cases that represent the agent's most common and important tasks.
  2. Measure Action Completion (AC): This is your primary success metric. It's a simple question: did the agent successfully achieve the user's end-to-end goal in the scenario?
  3. Measure Tool Selection Quality (TSQ): This is your primary efficiency metric. Did the agent use the correct tools with the correct parameters, and did it avoid unnecessary calls?
  4. Track Latency and Cost: For each scenario, record how long the agent took and how much it cost (e.g., token usage).
  5. Gate your deployment pipeline: Don't deploy if success metrics drop.

Basic Safety Testing

A focused set of tests for the most likely security and safety risks identified during the Discover phase.

When to use

Before the first production deployment.

MVP Steps
  1. Prompt Injection Tests: Use a library of known prompt injection attacks to test if the agent can be forced to ignore its instructions.
  2. Data Leakage Tests: If the agent handles sensitive data, create tests to see if you can trick it into revealing that data.
  3. Validate Guardrails: Test the input and output validators you built in the "Make" phase to ensure they work as expected.
Outputs

A safety scorecard showing pass/fail for the essential safety checks.

Pitfalls to Avoid
  • Treating safety as an advisory check
  • Not having automated safety tests that can be run regularly

Quick Reference: The MVP Mindset

  • Start with the smallest, most specific problem
  • Choose the simplest architecture that works
  • If the agent needs knowledge, use basic RAG
  • Build a stable skeleton with logging and kill switch
  • Prove its value with Action Completion (AC) and Tool Selection Quality (TSQ)
